Zero Trust Layer Protocol

ZTLP is a secure, identity-first network overlay that enforces cryptographic authentication before any network state is allocated. Invalid packets are rejected in a three-layer pipeline — Magic check, SessionID allowlist lookup, and HeaderAuthTag AEAD verification — making volumetric DDoS attacks structurally ineffective. ZTLP rides over existing IPv4/IPv6 infrastructure and is invisible to any node that does not possess a valid, hardware-backed cryptographic identity.

1. Problem Statement

The Internet was not designed to be secure. It was designed to be resilient and reachable — to move packets from one place to another regardless of who sent them or why. That was the right decision in 1974. It is the source of most of our problems in 2026.

1.1 The Cost of the Status Quo

  • DDoS attacks cost the global economy an estimated $2.5 billion annually. Volumetric attacks regularly exceed 1 Tbps. They work because the Internet accepts any packet from anyone.
  • Ransomware on critical infrastructure succeeds because internal networks are flat and unauthenticated. Once inside, lateral movement is trivial — the network has no concept of identity.
  • BGP route hijacking allows nation-state actors to silently redirect Internet traffic. BGP itself has no cryptographic path validation.
  • IoT insecurity is systemic. Billions of devices connect with no meaningful authentication, becoming permanent botnet members.
  • Compliance frameworks (HIPAA, PCI-DSS, CMMC, SOC 2) are compensating controls, not solutions. They detect and respond to threats that a secure-by-design network layer would never allow.

1.2 Why IPv6 Did Not Solve This

IPv6 solved address exhaustion. It did not change the fundamental model: any host can still send packets to any reachable address. Authentication and encryption are still optional. DDoS attacks work identically over IPv6. The root cause is philosophical — IPv4 and IPv6 treat the network as a neutral carrier of packets. ZTLP inverts this: the network refuses to carry packets that cannot prove their identity.

1.3 Why Now

The prerequisites for ZTLP now exist: hardware identity is mainstream (YubiKeys, TPM 2.0, Secure Enclaves); QUIC provides a proven encrypted UDP substrate; Zero Trust Architecture is industry consensus (NIST SP 800-207); the Noise Protocol Framework provides a proven handshake foundation; and the regulatory environment (CMMC 2.0, HIPAA, critical infrastructure mandates) demands action.

1.4 Who Benefits First

Healthcare, defense contractors (CMMC-regulated), MSPs, manufacturing/OT/ICS environments, and IoT deployments all have acute pain points that ZTLP directly addresses, making them natural early adopters.

1.5 The Vision

A network where ZTLP is widely deployed: port scanners return nothing; DDoS floods exhaust bandwidth but not CPU or memory; a stolen laptop cannot authenticate; ransomware cannot move laterally; connectivity follows identity, not location.

2. Introduction and Motivation

The Internet's current network layer was designed with reachability as the primary objective. Security was not a first-class design principle. As a result, the global Internet suffers from persistent structural problems:

  • Any host can send packets to any reachable address, regardless of authorization.
  • Services are exposed by default; security is enforced reactively by firewalls and ACLs.
  • DDoS attacks exploit the fact that routers cannot distinguish authorized from unauthorized traffic.
  • Encryption and authentication are optional add-ons, not protocol requirements.
  • Identity is tied to IP addresses, which are location-based and easily spoofed.

ZTLP proposes a different model. Rather than replacing IPv6, ZTLP defines a secure overlay that rides on existing Internet infrastructure and enforces a single rule before any network state is allocated:

"If you cannot prove who you are, you do not exist on this network."

ZTLP is not a replacement for the Internet. It is a protected lane on top of it.

4. Goals, Non-Goals, and Design Philosophy

4.1 Goals

  • Enforce cryptographic authentication before allocating any network state.
  • Eliminate 'open ports' — services are invisible without authorization.
  • Support hardware-backed identity (YubiKey, TPM, Secure Enclave) as a first-class primitive.
  • Provide performance-aware, multipath relay routing across the public Internet.
  • Deploy incrementally over existing IPv4/IPv6 infrastructure without router changes.
  • Define a distributed, DNS-like trust namespace (ZTLP-NS) for identity and service discovery.
  • Support gradual migration from legacy networks (Phase 1 through Phase 4).
  • Provide structural DDoS resistance through cryptographic air-gapping of the packet surface.

4.2 Non-Goals

  • ZTLP does not propose changes to IPv4/IPv6 headers or require Internet routers to be modified.
  • v0.9.1 does not implement hard bandwidth reservations across the backbone. Soft admission control via BANDWIDTH_HINT TLV is defined; hard reservations are a future revision target.
  • ZTLP is not a VPN product, though it shares architectural concepts with secure overlays.
  • ZTLP does not provide anonymity — identity is a core requirement.

4.3 Design Philosophy

Principle 1 — DDoS Resistance Must Be Structural, Not Reactive. ZTLP inverts the current model. Invalid packets are rejected through a three-layer pipeline ordered by cost. The Magic byte check costs a single comparison. The SessionID allowlist lookup costs an O(1) BPF hash map read. Only packets that survive both cheaper checks reach HeaderAuthTag AEAD verification. No session state is allocated until all three checks pass. The cost asymmetry strongly favors the defender.

Principle 2 — Identity Must Precede Communication. A node without a cryptographically verified identity cannot initiate communication. There are no open ports. There is no visible attack surface. The Zero Trust model is not enforced by policy on top of an insecure network — it is enforced by the protocol itself.

6. Protocol Overview

ZTLP operates as a secure overlay. The public Internet (IPv4/IPv6) acts as a dumb transport substrate. ZTLP does not require routers or ISPs to understand the protocol. A ZTLP packet is an encrypted, authenticated payload carried inside a standard UDP or QUIC datagram — indistinguishable from other encrypted UDP traffic.

5.1 Layering

[ Physical / Ethernet ]
[ IPv4 or IPv6 ]           ← public Internet routing
[ UDP or QUIC ]            ← transport substrate
[ ZTLP Header ]            ← identity, auth, routing
[ Encrypted Payload ]      ← application data

5.2 Connection Model

  1. Node discovers relay via bootstrap procedure.
  2. Node performs HELLO handshake with relay or peer.
  3. Mutual identity verification and policy check.
  4. Session keys derived; short-lived session established.
  5. Data packets flow; keys rotate automatically per session policy.
  6. Session closes via explicit CLOSE message or timeout.

5.3 Identity vs. Location

In IPv4/IPv6, an address is a location. In ZTLP, identity is primary. A Node ID is a stable identifier regardless of what IP the node currently uses, enabling seamless mobility, key rotation, and multihoming without breaking sessions.

6.0 Locator/Identifier Separation

ZTLP explicitly separates network identity from transport location. Three distinct layers:

  • Identifier layer — NodeID and ServiceID. Stable, permanent, hardware-backed. These never change and are what policies, ACLs, and logs reference.
  • Locator layer — IPv4/IPv6 transport addresses. Transient. Sessions are bound to identifiers, not locators, so locator changes do not break established sessions.
  • Routing layer — The relay mesh. Relay nodes dynamically select the best locator path for a given identifier, optimizing for latency, packet loss, congestion, and trust score.

DDoS attacks target locators — IP addresses. ZTLP's enforcement is identity-based. An attacker flooding a locator accomplishes nothing if packets cannot prove a valid identifier. The attack surface is the identifier space, not the locator space — and identifier space is cryptographically controlled.

7. ZTLP Addressing — Node Identity

6.1 Node ID

A ZTLP Node ID is a stable 128-bit random identifier assigned at node enrollment. It is NOT derived from the node's public key.

NodeID = RAND_128()   // generated once at enrollment, never changes

The NodeID represents the node as an entity — the device, service, or principal — independent of its current cryptographic key material. The public key is a separate attribute, bound to the NodeID via a signed ZTLP_KEY record in ZTLP-NS.

6.1.1 Why NodeID Is Not Derived From the Public Key

Early protocol designs (including HIP) derived node identifiers directly from public keys. This is cryptographically elegant but operationally painful. When NodeID = HASH(public_key), any key change — routine rotation, compromise response, lost YubiKey — changes the node's identity. Every policy, ACL, relay allowlist, and security log that referenced the old NodeID must be updated. In practice, key rotation is avoided, which is the opposite of good security hygiene.

ZTLP separates identity from key material. The NodeID identifies the node permanently. The public key proves current possession of the private key bound to that NodeID. This is consistent with modern identity systems: TLS certificates, SSH known hosts, OAuth identities, and Kubernetes service accounts all separate stable identity from rotating key material.

6.1.2 Key Binding and Rotation

A node's current public key is published as a ZTLP_KEY record in ZTLP-NS, signed by the zone authority. Key rotation is performed by publishing a new ZTLP_KEY record with an updated public key. The NodeID does not change. A node MAY have multiple active public keys simultaneously — useful for HSMs with backup keys or phased key rotation with overlap periods.

6.2 Service ID

ServiceID = TRUNCATE_128(SHA3-256(service-name || tenant-id))

rdp.clinic.acmedental.ztlp  →  ServiceID: 8f23:9c11:ae45:...
backup.server.example.ztlp  →  ServiceID: c341:0f8a:21bc:...

6.3 Human-Readable Names

The ZTLP-NS namespace maps human-readable names to Node IDs, Service IDs, relay addresses, and policy records — functioning similarly to DNS but with mandatory signing and delegated trust.

7.5 Payload

The payload is AEAD-encrypted application data. The authentication tag produced by the AEAD cipher is appended to the ciphertext. Decryption MUST fail and the packet MUST be silently dropped if authentication fails.

9. ZTLP-NS — Distributed Trust Namespace

ZTLP-NS is the control-plane identity and discovery layer, inspired by DNS delegation and DNSSEC chain-of-trust principles, adapted for ZTLP's identity-first model.

8.1 Design Principles

  • Hierarchical delegation — trust flows from root anchors through operator zones to individual nodes.
  • All records are signed — unsigned records MUST be rejected.
  • Federated roots — multiple trust roots are supported; no single global authority.
  • ZTLP-NS is a control-plane lookup layer — NOT used for per-packet routing decisions.

8.2 Namespace Structure

Root Trust Anchors (embedded in ZTLP software)
└── Operator Zones  (e.g., example.ztlp)
    └── Tenant Zones  (e.g., acmedental.example.ztlp)
        └── Node / Service Records

8.3 Record Types

ZTLP_RELAYRelay node: Node ID, IPv6 endpoints, capacity metrics.relay1.apac.ztlp → NodeID + endpoints

8.4 Federated Trust Roots

ZTLP-NS supports multiple trust roots. A node MAY trust multiple roots simultaneously. No single trust anchor SHALL be required for ZTLP network participation. Bootstrap discovery URLs MUST span at least three independent operators to prevent any single party from controlling network access.

10. Node Initialization and Bootstrap Procedure

A ZTLP node that has no prior session state MUST follow the Node Initialization Procedure (NIP). The following steps MUST be attempted in order:

Step 1 — HTTPS Discovery (REQUIRED)

The node MUST attempt an HTTPS GET to each hardcoded discovery URL. The response MUST be a signed JSON object containing relay addresses, Node IDs, public keys, and a validity timestamp. The node MUST validate the signature before using any relay address.

Step 2 — DNS-SRV Discovery (RECOMMENDED)

Query for SRV records at: _ztlprelay._udp.bootstrap.ztlp. If DNSSEC validation is available, the node MUST validate the response chain before use.

Step 3 — Hardcoded Trust Anchors (FALLBACK)

If Steps 1 and 2 both fail, the node MUST attempt connection to the hardcoded relay list embedded in the software distribution. This list MUST contain a minimum of 15 relay nodes, spanning at least 5 distinct ASNs and 3 geographic regions.

9.2 Post-Bootstrap Peer Exchange

Once a node establishes a session with any relay, it MUST request the relay's routing table via a PEER_EXCHANGE message. The relay responds with a signed list of known relay nodes.

9.3 Discovery URL Resilience

Implementations MUST include discovery URLs hosted by at least 3 independent operators, geographically and organizationally independent, preventing single-point-of-failure in HTTPS discovery.

11. Handshake and Session Establishment

10.1 Handshake Overview

ZTLP uses Noise Protocol Framework pattern (Noise_XX) for session establishment, providing: mutual authentication, perfect forward secrecy, and identity hiding (identities are encrypted after the first message).

10.2 Message Flow

Initiator                          Responder
|                                   |
|--- HELLO (ephemeral pubkey) -----> |
|                                   |
|<-- HELLO_ACK (ephemeral pubkey,   |
|    encrypted identity + cert) ---- |
|                                   |
|--- (encrypted identity + cert,    |
|    policy request) -------------> |
|                                   |
|<-- SESSION_OPEN (SessionID,       |
|    PolicyTag, session keys) ------ |
|                                   |
|=== DATA flows (encrypted) ======= |

10.3 Policy Enforcement

The Responder MUST perform the following checks before issuing SESSION_OPEN: verify certificate chain against a trusted ZTLP-NS root; check revocation status; evaluate policy (is SrcNodeID authorized to reach DstSvcID?); check device posture if required by policy. Reject with ERROR if any check fails — no state is allocated on failure.

10.4 Key Rotation

Session keys MUST be rotated at intervals defined by the CryptoSuite policy, or upon explicit REKEY message. A new SessionID is issued on each rotation. Applications MUST NOT experience connection interruption during key rotation.

12. Relay Node Architecture

11.1 Role of Relay Nodes

ZTLP Relay nodes forward authenticated ZTLP traffic between nodes, participate in peer exchange, and publish capacity metrics (latency, packet loss, bandwidth) to ZTLP-NS.

11.2 Relay Selection — Two-Step Model

PathScore = w1*Latency + w2*PacketLoss + w3*Congestion + w4*TrustScore

TrustScore is a relay-operator reputation metric derived from: the relay's published ZTLP_RELAY record (signed by a trusted operator authority), uptime and availability history, and whether the relay participates in an authenticated relay federation (Section 12 — federated relays receive higher baseline TrustScore than unenrolled relays). Implementations MAY extend TrustScore with operator-signed attestations. Lower PathScore is preferred.

PathScore-based selection alone causes hotspots at scale. ZTLP uses a two-step model:

  1. Step 1 — Candidate set via consistent hash: Map the destination ServiceID to the nearest N relay nodes on a consistent hash ring of relay NodeIDs: CandidateSet = Hash(ServiceID) → nearest 3 relays. This provides stable, load-distributed assignment.
  2. Step 2 — Optimize within the candidate set: Select the relay with the lowest PathScore. This provides latency optimization within the bounded set without global convergence on a single best relay.

Consistent hashing ensures traffic to popular services is automatically distributed; relay churn causes minimal session migration; the mesh scales to thousands of nodes without central coordination.

11.3 Relay Operator Requirements

Relay nodes MUST: maintain a valid ZTLP_RELAY record; publish latency and capacity metrics at least once per 60 seconds; support all four transport modes (defined in Section 14); honor CLOSE messages and release session state within 30 seconds; and NOT log plaintext payload content.

11.4 Ingress Admission Domains and Relay Admission Tokens

ZTLP relay deployments SHOULD organize relays into Ingress Admission Domains. Relay roles:

  • Ingress Relay — handles first-contact traffic, HELLO messages, Stateless Admission Challenge, and issues Relay Admission Tokens (RATs) upon successful authentication.
  • Transit Relay — forwards authenticated sessions. SHOULD only accept traffic with valid RATs or already-authenticated sessions.
  • Service Relay — final-hop forwarding near destination services. SHOULD reject first-contact HELLOs not from trusted ingress relays.

Deterministic ingress assignment via consistent hashing: IngressSet = Hash(NodeID || bootstrap_salt) → nearest N relays (N = 3 recommended). The bootstrap_salt is periodically rotated to prevent long-term fingerprinting.

A Relay Admission Token (RAT) MUST contain: NodeID, issuing relay NodeID, issuance timestamp, expiration timestamp, and a cryptographic signature. Tokens SHOULD have lifetimes of 5–30 minutes. Token verification MUST be cheaper than full Noise_XX handshake processing.

11.5 Authenticated Relay Federation and Capacity Classes

Relay operators MAY form authenticated relay federations. Each participating operator MUST possess an Operator Identity published as a ZTLP_OPERATOR record in ZTLP-NS.

Capacity classes for federated relays:

  • C0 (Best Effort) — ZTLP traffic over public Internet with no special treatment.
  • C1 (Authenticated Relay Class) — relay-to-relay traffic authenticated by federation membership.
  • C2 (Soft Reserved Capacity) — operators reserve a portion of link capacity for relay traffic during congestion.
  • C3 (Hard Reserved Capacity) — dedicated bandwidth allocation between relay domains.

13. Hardware Enforcement Profiles

Four profiles push ZTLP enforcement progressively closer to the wire. All profiles feed decisions from the ZTLP control plane; no profile requires the switch or NIC to independently validate full cryptographic identity.

Profile 1 — Software Enforcement (eBPF/XDP)

The baseline enforcement model, implementable today on any Linux host. XDP programs attach directly to the NIC driver and execute before the Linux kernel stack. The XDP pipeline:

1. Read first 16 bits. If Magic != 0x5A37 → XDP_DROP (immediate, zero state)
2. Read SessionID field. Check against eBPF SessionID allowlist map.
   If SessionID not in map AND MsgType != HELLO → XDP_DROP
3. Rate-limit HELLO packets per source IP (token bucket in eBPF map).
   If rate exceeded → XDP_DROP
4. Pass to kernel → full HeaderAuthTag verification in ZTLP userspace daemon.

Profile 2 — SmartNIC Offload (DPU)

Moves full ZTLP HeaderAuthTag verification off the host CPU onto the DPU (e.g., Nvidia BlueField-3, AMD Pensando). The DPU Offload Agent maintains session key table, anti-replay window, and HELLO rate limiter. Only packets passing all checks are forwarded to the host PCIe bus — attack traffic never reaches host memory. RECOMMENDED for relay nodes and gateway nodes handling high-volume traffic.

Profile 3 — Programmable Switch Enforcement (P4)

P4-capable switch hardware enforces ZTLP at full line rate. Two-tier split:

  • Tier 1 — Switch ASIC (P4, line rate): Magic byte check, SessionID allowlist lookup, source IP rate limiting, HELLO packet metering. Failing packets dropped at the switch with no server involvement.
  • Tier 2 — Server (eBPF or DPU): Full HeaderAuthTag AEAD verification. Only packets surviving Tier 1 reach this layer.

The ZTLP control plane populates P4 match-action tables via P4Runtime over gRPC. A revoked identity is removed from the switch table within one control plane cycle (typically under one second).

Profile 4 — Native Switch ASIC (Future)

Long-term target: switch ASICs natively understanding ZTLP packet structure without P4 programming — analogous to how modern switches natively implement MACsec (IEEE 802.1AE). Full on-chip HeaderAuthTag AEAD verification is an OPTIONAL capability for Profile 4 hardware.

14. Transport Fallback and NAT Traversal

ZTLP MUST function in environments where ISPs, enterprise firewalls, or carrier-grade NAT block or reshape UDP traffic. Implementations MUST support the following transport fallback ladder in order:

PriorityTransportConditionNotes
1UDP / ZTLP PortPreferredNative ZTLP transport. Best performance.
2UDP / 443If port blockedHarder for ISPs to block without disrupting QUIC.
3TCP / 443 (TLS framed)If UDP blockedZTLP framed inside TLS record layer.
4WebSocket over HTTPSLast resortMaximum firewall compatibility. Higher overhead.

For direct peer-to-peer connections (without relay), ZTLP nodes SHOULD attempt UDP hole-punching coordinated via a relay node. If hole-punching fails within 5 seconds, the relay MUST continue to forward the session.

15. Routing, Congestion Control, and Multipath

13.0 Congestion Control

ZTLP sessions MUST implement a congestion control algorithm. A ZTLP implementation without congestion control risks starving competing TCP flows and contributing to congestion collapse. Implementations SHOULD use BBR or CUBIC as the default. A ZTLP-native algorithm optimized for multi-relay paths is listed as an open issue in Section 21.

13.0b Path MTU Discovery and Fragmentation

The ZTLP base header is 64 bytes. Combined with UDP (8 bytes) and IPv6 (40 bytes), minimum overhead before payload is 112 bytes. Implementations MUST perform path MTU discovery using standard ICMP/ICMPv6 "Packet Too Big" and support ZTLP-layer fragmentation and reassembly.

MaxPayload = PathMTU - 112 - ExtensionHeaderBytes

Implementations MUST NOT rely on IP fragmentation as a substitute for ZTLP-layer fragmentation, as IP fragments bypass the ZTLP header authentication check at the receiver.

Multipath

The MULTIPATH flag signals a session uses multipath mode. ZTLP nodes MAY split flows across multiple relay paths for resilience, aggregate bandwidth, and geographic traffic distribution. Receivers MUST reassemble using PacketSeq. Anti-replay window for MULTIPATH sessions MUST be at least 1024 packets.

13.2 Mobility

Because Node IDs are identity-based and not location-based, a node that changes its underlying IPv6 address MUST issue a MIGRATE message containing its new transport endpoint. The existing session continues without interruption. Applications MUST NOT observe a connection break during migration.

16. Key Management and Hardware Identity

15.1 Hardware-Backed Keys

ZTLP STRONGLY RECOMMENDS long-term private keys be stored in hardware security devices: FIDO2 / YubiKey (USB, NFC), Trusted Platform Module (TPM 2.0), Secure Enclave (Apple Silicon, ARM TrustZone), and HSM for relay/gateway nodes. When hardware identity is in use, the private key MUST NOT be exportable. Signing operations MUST occur on-device.

15.2 Certificate Lifecycle

Every ZTLP node MUST have a certificate issued by a ZTLP-NS trust root. The lifecycle: enrollment, issuance, renewal, and revocation. Certificate TTL SHOULD NOT exceed 90 days. Short-lived session tokens SHOULD NOT exceed 24 hours.

15.3 Node Identity Assurance Model and Attestation

ZTLP defines four assurance levels for Node identities:

  • A0 (Software Identity) — private key stored in software; suitable for development or low-risk services.
  • A1 (Hardware-Backed Identity) — private key stored in a hardware security device (TPM, Secure Enclave, FIDO2 token); key extraction is not possible.
  • A2 (Attested Managed Device) — hardware-backed key plus device posture verification (TPM attestation, enterprise MDM enrollment).
  • A3 (Infrastructure Identity) — high-assurance identity for relay nodes, gateways, and infrastructure services; keys stored in an HSM.

Assurance attributes are published in ZTLP-NS using an optional ZTLP_ATTEST record, containing NodeID, AssuranceLevel, AttestationType, IssuingAuthority, validity timestamps, and a Signature by the attestation authority. Nodes without attestation records are treated as Assurance Level A0.

ZTLP distinguishes three separate concepts: Identity (the NodeID — "Who is this node?"), Proof (cryptographic evidence of key control — "Can this node demonstrate control?"), and Assurance (verified properties of the device environment — "How trustworthy is this environment?").

18. Security Considerations

16.1 DDoS Resistance — Three-Layer Pipeline

ZTLP's primary DDoS defense is structural. Each layer is ordered by computational cost:

  • Layer 1 — Magic byte check (nanoseconds, no crypto): A single 16-bit comparison at the NIC driver via XDP. Any packet not beginning with 0x5A37 is dropped before the kernel sees it. Eliminates all random UDP flood traffic.
  • Layer 2 — SessionID allowlist lookup (microseconds, no crypto): An O(1) BPF hash map lookup against active SessionIDs. Packets with unknown SessionIDs are rate-limited and dropped before any cryptographic work. Eliminates attackers who know the ZTLP format but do not know active session identifiers.
  • Layer 3 — HeaderAuthTag AEAD verification (real cryptographic cost): ChaCha20-Poly1305 or AES-GCM tag verification only on packets that passed both Layer 1 and Layer 2. A packet reaching Layer 3 has already demonstrated it knows a valid active SessionID — meaning the attacker must have compromised session state rather than simply generating flood traffic. Session state is not allocated until Layer 3 passes.

16.2 Amplification Prevention

ZTLP nodes MUST NOT generate responses whose byte size exceeds the initiating packet before authentication is complete. HELLO responses and CHALLENGE messages MUST be padded or truncated such that response_bytes ≤ request_bytes. This prevents UDP amplification attacks analogous to DNS/NTP/SSDP reflection.

16.3 Stateless Admission Challenge (SAC)

To defend against handshake exhaustion by botnets, ZTLP relays and services MAY require unknown initiators to complete a Stateless Admission Challenge prior to full session establishment:

Initiator                    Responder

HELLO              -------->
                   <--------  CHALLENGE (nonce, difficulty, expiry, cookie)
HELLO_PROOF        -------->  (solution + original HELLO)
                   <--------  HELLO_ACK (proceed with Noise_XX)

The CHALLENGE is stateless on the responder side. The cookie is a MAC over (source_ip, timestamp, responder_secret), verifiable when HELLO_PROOF arrives without any per-client state stored — directly analogous to TCP SYN cookies and QUIC retry tokens. Puzzle difficulty is adaptive: zero under normal load, increasing automatically as HELLO rate exceeds configurable thresholds.

SAC MUST NOT be applied to established sessions, rekeying operations, trusted relay-to-relay handshakes, or nodes presenting valid short-lived trust tokens from a prior successful session.

16.4 Clock Synchronization

The Timestamp field is a replay heuristic, not a hard rejection criterion. Implementations MUST NOT reject packets solely based on timestamp deviation. Implementations SHOULD flag packets with timestamps more than ±5 minutes from local time for additional scrutiny, but MUST NOT drop them on that basis alone.

Cryptographic Agility

The CryptoSuite field allows future algorithm upgrades without protocol version changes. Implementations MUST NOT negotiate CryptoSuites with known weaknesses.

Revocation Latency

ZTLP_REVOKE records propagate through ZTLP-NS with TTL-governed latency. Implementations MUST check revocation status during session establishment. For high-security deployments, short certificate TTLs are preferred over reliance on revocation propagation.

19. Privacy Considerations

18.1 Packet-Level Identity Privacy

NodeID values represent long-lived identities and MUST NOT appear in established data packets. After SESSION_OK, the data-plane header contains only the canonical 96-bit SessionID — no SrcNodeID, no DstSvcID. This is the same model used by QUIC (Connection ID), WireGuard (Session Index), and MPLS (Label): identity is verified once at handshake time, then routing proceeds by a compact, opaque session token.

The SessionID provides per-session unlinkability because it is assigned with cryptographically strong entropy during the Noise_XX handshake and rotated on every REKEY. A passive observer — an ISP, backbone router, or hostile network operator — observing ZTLP data packets sees only a random-looking 96-bit value that changes every session. No stable handle exists for cross-session correlation. NodeID values are transmitted only during the authenticated handshake phase, which already provides identity hiding via the Noise_XX pattern (identities are encrypted after the first message).

Implementations MUST NOT include SrcNodeID or DstSvcID in established data packets. Such packets are non-conforming with respect to this section and expose long-lived identifiers to passive observers.

18.2 Remaining Privacy Risks

  • ZTLP-NS lookups may reveal organizational relationships to namespace operators. Implementations SHOULD use oblivious lookup mechanisms where available.
  • Traffic volume and timing patterns can enable inference attacks even with pseudonymous identifiers.
  • The handshake phase necessarily reveals that ZTLP is in use, though not the specific identities after the first message.
  • Relay operators can observe which SessionIDs are active, even if they cannot identify underlying NodeIDs.

18.3 Operational Guidance

ZTLP-NS operators SHOULD NOT log individual lookup queries beyond what is required for operational purposes. Relay operators MUST NOT log plaintext payload content. ZTLP is not an anonymity protocol. Applications requiring strong anonymity SHOULD use additional layers such as onion routing above ZTLP.

23. Trust and Authority Model

ZTLP separates trust anchoring, identity enrollment, policy issuance, and revocation into four distinct logical authority roles:

22.1 Trust Roots

Trust Roots are the top-level anchors for ZTLP identity verification, responsible for defining root certificates or public keys, delegating authority to Enrollment and Operator Authorities, and defining trust boundaries. ZTLP does not require a single global root authority — multiple independent trust roots may coexist.

22.2 Enrollment Authorities

Enrollment Authorities bind cryptographic identities to NodeIDs by issuing signed records that bind: NodeID ↔ Public Key ↔ Organizational Identity. These bindings are published in ZTLP-NS using ZTLP_KEY records. Enrollment Authorities MUST sign issued records using keys that chain to a recognized Trust Root.

22.3 Policy Authorities

Policy Authorities define access control rules governing service communication, publishing signed ZTLP_POLICY records that specify which identities or identity classes are permitted to establish sessions with a given service.

22.4 Revocation Authorities

Revocation Authorities invalidate previously issued identity bindings using ZTLP_REVOKE records. ZTLP nodes MUST periodically synchronize revocation records from ZTLP-NS.

22.5–22.7 Delegation, Conflict Resolution, and Rationale

Trust Roots may delegate authority through signed delegation records. When conflicting records are encountered: Revocation records override all other record types; enterprise root records take precedence over public roots; records from the issuing authority take precedence over third-party delegations. Separating trust anchoring, enrollment, policy, and revocation reduces systemic risk and enables flexible deployment models.

24. Public Internet Interoperability and Overlay Architecture

ZTLP operates as a secure identity overlay above the public Internet. Internet routers do not need to understand ZTLP. ZTLP packets are encapsulated inside normal UDP or TCP traffic and forwarded without modification.

23.1 Layering Model

Application / ZTLP Transport / UDP or TCP / IPv4 or IPv6 / Public Internet

This overlay model follows the same deployment precedent as WireGuard (over UDP), QUIC (over UDP), Tailscale, Cloudflare Warp, and Tor. Overlay protocols succeed because they deploy immediately on existing infrastructure, whereas replacing Internet routing layers has historically taken decades.

23.2 Legacy Compatibility and Incremental Adoption

ZTLP can coexist with legacy networking. A service may simultaneously support standard Internet access and ZTLP-secured access. ZTLP Gateways allow existing internal networks to participate in ZTLP without modifying every host. The Internet provides packet transport; ZTLP provides the identity and authorization layer.

25. Commercial Adoption and Gateway Deployment Model

ZTLP MUST support gateway-based deployment for existing services. Large organizations will not rewrite their infrastructure to adopt a new protocol. ZTLP achieves adoption by sitting in front of existing systems via an edge gateway, leaving internal infrastructure unchanged.

24.1 Gateway Architecture

A ZTLP Edge Gateway is deployed analogously to a load balancer, API gateway, or reverse proxy. It performs ZTLP handshake, identity validation, policy enforcement, session creation, and traffic forwarding to internal services. Internally, existing infrastructure continues using HTTP, TLS, gRPC, database protocols, and microservices without modification.

Client / Internet / ZTLP Relay / ZTLP Edge Gateway / Internal Service Infrastructure

24.2 Node Identity Types for Commercial Services

For commercial adoption, ZTLP MUST support: user nodes, device nodes, service nodes, relay nodes, and gateway nodes. Service-to-service authentication is a primary requirement for cloud providers operating microservice architectures, B2B APIs, and partner integrations.

24.3 Commercial Value Proposition

  • Fraud and abuse reduction — identity-first admission control prevents bot scraping, credential stuffing, and abusive API usage.
  • Structural DDoS cost reduction — admission control drops unauthorized packets before they reach service infrastructure.
  • Trusted device and service access — ZTLP identities enable secure B2B APIs and partner integrations.
  • Regulatory compliance — the identity binding model supports audit trails required in financial, healthcare, and government environments.

24.4–24.5 DNS Discovery and Phased Adoption

A DNS record of the form _ztlp.api.service.example.com returns the service NodeID, relay endpoint, and policy requirements. ZTLP-aware clients connect automatically; legacy clients fall back to standard HTTPS. Phased adoption: Phase 1 — protect sensitive APIs; Phase 2 — administrative interfaces; Phase 3 — customer authentication endpoints; Phase 4 — internal east-west traffic.

26. Policy-Constrained Adaptive Path Selection

ZTLP implementations SHOULD evaluate direct, gateway-mediated, and relay-assisted paths as candidate transport paths. Direct transport is not inherently preferred over relay-assisted transport — a relayed path MAY be selected when it provides lower effective cost or better stability.

25.2 Path Scoring

PathScore = w1 * Latency + w2 * PacketLoss + w3 * Congestion + w4 * TrustScore

All candidate paths are evaluated as peers. The path with the best effective score subject to policy constraints is selected.

25.3 Relay-Optimized Routing

A direct Internet path may be topologically shorter but operationally worse due to poor intercontinental peering, congestion, or asymmetric routing. For example, a direct path from China to the United States may have substantially higher latency and packet loss compared to a relay-assisted path routing through Singapore. ZTLP path selection treats relay-assisted paths as first-class candidates evaluated on measured network conditions, not as fallback mechanisms.

25.4–25.5 Continuous Monitoring and Session Migration

Implementations SHOULD continuously monitor active and standby paths and MAY migrate established sessions to a better path when quality materially improves or the current path degrades. Session migration MUST preserve session identity and cryptographic state. Automatic path switchover is transparent to applications.

27. Handshake Flood Resistance

ZTLP assumes session establishment mechanisms may be targeted by adversaries attempting to exhaust relay resources. Four mechanisms applied in sequence:

  1. Admission domains — relays only accept handshake attempts assigned to them via consistent hashing, distributing handshake load across the relay mesh so no single relay can be overwhelmed.
  2. Cheap early rejection — the three-stage validation pipeline (magic byte check, session lookup, HeaderAuthTag verification) rejects unauthenticated traffic with minimal CPU cost before any relay state is allocated.
  3. Proof-of-work puzzles — during session establishment, relays MAY issue computational puzzles that clients must solve before the relay commits resources to the session. Mirrors the SYN cookie defense model.
  4. Stateless early rejection — relays do not allocate session state until the client has demonstrated proof of work and passed initial validation.

ZTLP ends up more resistant to DDoS than the current Internet model: today anyone may send unlimited packets to any public IP, whereas ZTLP requires demonstrated computational effort before any session resource is committed.

28. Identity-Gated Connectivity: No-Open-Port Networking

ZTLP enables a network model in which services operate with no publicly exposed ports. A service protected by ZTLP does not listen on any publicly reachable socket. It accepts traffic exclusively through authenticated ZTLP sessions. To an unauthenticated observer, the service is invisible: it does not respond to port scans, does not acknowledge connection attempts, and generates no observable attack surface.

ZTLP makes services unreachable until identity is proven.

27.2 What No-Open-Port Networking Eliminates

  • Port scanning and Shodan-style discovery — services do not respond to unauthenticated probes.
  • Botnet probing — attackers cannot send packets directly to services.
  • Many DDoS vectors — services never see anonymous traffic.
  • Exploit scanning — tools that search for vulnerabilities on open ports find no surface to target.

27.3 Example: ZTLP-Protected SSH

Today, an SSH server listens on TCP port 22, visible to any scanner. With ZTLP:

ztlp connect ssh.example

The server has no publicly listening port. Only authenticated ZTLP sessions can reach it. The server is invisible to the Internet until identity is proven.

27.4 Comparison with Existing Systems

Existing systems implement partial versions of this idea. Tailscale requires a central control plane. ZeroTier still exposes node addresses. Tor hidden services are specialized and slow. Cloudflare Access is HTTP-only. VPNs still expose gateways. ZTLP provides a general-purpose transport-level solution where identity, not IP address, determines reachability. The key shift: IP address = reachable becomes identity = reachable.

29. SessionID as Primary Routing Primitive

Once authenticated, traffic is routed using a compact SessionID that allows relays and gateways to forward packets without repeating identity verification on every packet. This separation between identity validation and packet forwarding enables high-performance routing while preserving strong authentication guarantees.

The architecture separates into three layers: Identity Layer (authentication during handshake only), Session Layer (routing via SessionID), and Transport Layer (encrypted packet delivery). This follows the same pattern used by QUIC (Connection ID), WireGuard (Session Index), MPLS (Label), VXLAN (VNI), and IPsec (SPI).

28.1 Three-Phase Operation

  1. Phase 1 (Identity) — the client proves identity using NodeID and public key.
  2. Phase 2 (Session Creation) — the relay or gateway generates a SessionID (96-bit random value) representing the established authenticated connection.
  3. Phase 3 (Packet Transport) — all subsequent packets carry only SessionID, sequence number, authentication tag, and payload. Relays read the magic byte, look up the SessionID, and forward. This enables MPLS-like hardware acceleration of the data path.

28.2 Canonical Relay Forwarding Procedure

The following is the normative fast-path forwarding procedure for relay nodes handling established data packets:

1. Read first 16 bits. If Magic != 0x5A37 → DROP (no state, no log)
2. Read SessionID (96 bits). Lookup in session table.
   If not found → DROP (no state, no log)
3. Validate HeaderAuthTag (AEAD). If invalid → silent DROP
4. Update anti-replay window for PacketSeq.
   If replay detected → silent DROP
5. Lookup next-hop for SessionID → forward packet.

NodeID is NOT consulted on the forwarding path. The SessionID is the sole routing key. This is the ZTLP fast path and MUST be the only code executed per packet on established sessions.

28.3 Plane Separation

PlanePurposeIdentifiers Used
Admission PlaneHandshake, authentication, policy check, session creationNodeID, ServiceID, PolicyTag, certificates
Forwarding PlanePer-packet routing and relay forwardingSessionID only
Service PlaneGateway delivery to application infrastructureSessionID → internal service mapping

Admission plane operations are expensive and occur once per session. Forwarding plane operations are cheap and occur on every packet. This separation is the architectural foundation of ZTLP's DDoS resistance and high-performance routing.

28.3 Session Migration Using SessionID Stability

Because the SessionID is the sole routing key for data packets, path switching does not require session teardown. When a better relay path is available, the SessionID remains constant while the underlying transport path changes. Clients do not need to reconnect. Applications do not observe the migration.

31. Historical Context: Identity Networking and Protocol Precedents

The concept of identity-structured networking is not new. Novell's IPX/SPX attempted a partially identity-like addressing model. Understanding why it failed clarifies why ZTLP is designed differently and why the present technological environment makes identity-first transport viable now.

30.1 IPX/SPX: What It Did and Why It Failed

Novell IPX addresses had a structure of NetworkID:NodeID:Socket, where NodeID was bound to the device MAC address. It worked well in controlled corporate LANs but failed at Internet scale for three reasons: addresses could be spoofed (no cryptographic binding), service discovery relied on broadcast (impractical globally), and TCP/IP won standardization.

30.2 How ZTLP Solves What IPX Could Not

  • Address spoofing — ZTLP uses cryptographic NodeIDs with public key binding, making identity forgery computationally infeasible.
  • Broadcast discovery — ZTLP uses ZTLP-NS and relay-based discovery, which scales globally without broadcast.
  • Deployment — ZTLP operates as an overlay over existing TCP/IP infrastructure, requiring no router changes.

30.3 The Broader Historical Pattern

Networking ideas often fail once and succeed decades later when cryptographic primitives, global infrastructure, and deployment models catch up. Frame Relay evolved into SD-WAN. ATM influenced MPLS. Telnet became SSH. ZTLP represents the convergence of identity-structured networking (IPX), relay routing (Tor), cryptographic sessions (WireGuard), and session-based transport (QUIC) into a unified overlay architecture.

32. Practical Deployment Path and Reference Implementation

ZTLP can be deployed immediately on today's Internet without waiting for operating system or router support. The protocol runs over UDP on standard Internet infrastructure, following the same deployment model as QUIC, WireGuard, Tailscale, and ZeroTier.

31.1 User-Space Deployment Stack

Application / ZTLP (user-space) / UDP / IPv4 or IPv6 / Internet

Four reference implementation components: ztlp-client (Linux, Windows, macOS, mobile); ztlp-relay (validates sessions, forwards packets by SessionID); ztlp-gateway (deploys in front of existing services); ztlp-service (a protected endpoint with no public ports).

31.2 Handshake Flow

Client                    Relay/Gateway
|                            |
|------- HELLO (NodeID) ------>|
|<-- CHALLENGE (puzzle+nonce) --|  [magic check + session lookup]

31.3 NAT Traversal

31.4 Path to Kernel and Hardware Acceleration

Kernel integration, eBPF acceleration, and SmartNIC support are optional enhancements that may follow once the protocol proves useful in user-space deployments. QUIC operated in user-space for years before receiving OS-level support. ZTLP follows the same trajectory: user-space first, kernel acceleration later.
